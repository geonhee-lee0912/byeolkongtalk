# 이번달 운세(monthly) 내용 리뉴얼 — 설계 문서

**작성일:** 2026-06-06
**대상:** 별콩 운세 `monthly`("이번달 어떤 일들이?", 15별)

## 목표

1. monthly 리포트를 줄글 마크다운 → **구조화 JSON 카드**로 전환 (daily 파이프라인의 월간 버전). daily 기반 위에 월간 전용 요소(월건 풀이·주차별 흐름·주목할 시기·이번 달 테마)를 얹어 내용을 풍부하게.
2. **같은 달 재방문 처리**: 자동 리다이렉트가 아니라, 월간 사주 선택 화면에서 이미 이번 달 리딩이 있는 프로필을 고르면 CTA를 "이 사주로 운세 보기"(결제) → "이번 달 운세 다시보기"(무결제, 기존 리딩으로 이동)로 바꾼다.

## 비목표 (Out of Scope)

- saju_full(2026 사주분석)·daily·tarot 의 내용/포맷 변경.
- 정확한 "행운의 날짜" 계산 — `DailyLuck` 에 품질 점수가 없어 false precision 위험. 주목할 시기는 정성적(상순/중순/하순)으로만.
- 월간을 "한 달 1회 고정"으로 강제하거나 재구매를 막는 것 — 결제는 그대로 가능, 단지 기존 리딩을 다시보기로 안내.

## 1. 데이터 모델 — `lib/fortune/monthly-report.ts` (신규)

daily-report.ts 와 동일한 구조·함수 시그니처를 평행하게 둔다.

```ts
export type MonthlySectionKey = "money" | "work" | "love" | "health" | "study";

export interface MonthlyWeek {
  week: 1 | 2 | 3 | 4;
  body: string;
}

// AI 가 생성하는 부분
export interface MonthlyReportAI {
  stars: number;              // 1~5 (이번 달 종합운)
  theme: string;              // 이번 달 테마 한 줄 (~20자)
  summary: string;            // 한 줄 총평 (~30자)
  lucky: { keyword: string; color: string };  // (daily 의 time 은 월간에 부적합 → 제거)
  intro: string;              // 월건 두 글자 풀이
  weekly: MonthlyWeek[];      // 1~4주 고정 4개
  sections: { key: MonthlySectionKey; body: string }[];  // daily 와 동일 5개 도메인
  timing: { good: string; caution: string };  // 주목할 시기 (정성적)
  balance: { good: string; warn: string };    // 이번 달 챙길 점
  note: string;               // 별콩이 한마디
}

// 서버가 병합하는 월건(결정론적) — temporal.month 출처
export interface MonthlyWolgeon {
  stem: string;        // "갑"
  branch: string;      // "오"
  hanja: string;       // "甲午"
  stemElement: FiveElement;
  branchElement: FiveElement;
}

// 저장/렌더 최종 형태
export interface MonthlyReport extends MonthlyReportAI {
  v: 1;
  wolgeon: MonthlyWolgeon;
}
```

**함수 (daily-report 와 동형):**
- `parseMonthlyReportJson(raw): MonthlyReportAI | null` — 코드펜스/잡텍스트 허용, 첫 `{`~마지막 `}` 파싱. 필수 필드 검증: stars(1~5 클램프), theme/summary/intro/note(비어있지 않음), lucky{keyword,color}, weekly(week 1~4 각 body 존재, 4개 모두), sections(5개 도메인 모두 존재, 표준 순서 정렬), timing{good,caution}, balance{good,warn}. 하나라도 누락 시 null.
- `buildMonthlyReport(ai, temporal): MonthlyReport` — `temporal.month`(PillarLite)에서 월건 글자/오행 병합.
- `serializeMonthlyReport(report): string`
- `tryParseStoredMonthlyReport(content): MonthlyReport | null` — `{`로 시작 + `v===1` + `wolgeon` 객체 + `weekly` 배열 확인. legacy 마크다운이면 null.

**도메인 메타:** daily 와 동일한 5개(💰재물 / 💼직장·업무 / 💗애정·관계 / 🌿건강·멘탈 / 📄학업·문서·이동). `DAILY_SECTIONS` 를 재사용한다(같은 도메인 집합).

## 2. 공유 모듈 — `lib/fortune/element.ts` (신규, 소규모 추출)

daily-report.ts 에 있는 오행 관련 상수를 추출해 daily·monthly 양쪽에서 import:
- `ELEMENT_COLOR: Record<FiveElement, string>` (오행 → 색)
- `STEM_ELEMENT` / `BRANCH_ELEMENT` (천간·지지 한글 → 오행)
- 헬퍼 `stemToElement(stem)` / `branchToElement(branch)`

daily-report.ts 는 기존 import 호환을 위해 `ELEMENT_COLOR` 를 **re-export** 한다(DailyReportCard 등 기존 사용처 무변경).

## 3. 프롬프트 — `lib/fortune/prompt.ts`

`SECTION_GUIDE.monthly` 를 마크다운 `##` 4섹션 → **JSON 출력 1개** 지시로 교체 (daily 방식 그대로). 주입 토큰:
- `{{THIS_MONTH}}` — "2026년 6월" (기존 존재)
- `{{THIS_MONTH_PILLAR}}` — 이번 달 월건 두 글자. `buildFortuneSystem` 에서 `input.saju?.temporal?.month.stem + branch` 로 치환 (없으면 "이번 달 월건").

JSON 스키마(요지):
```
{
  "stars": <1~5 정수, 월건과 일간 조화 정직 반영>,
  "theme": "<이번 달 테마 한 줄, ~20자>",
  "summary": "<한 줄 총평, ~30자>",
  "lucky": { "keyword": "<키워드 한 단어>", "color": "<행운 색 이름>" },
  "intro": "<월건({{THIS_MONTH_PILLAR}}) 두 글자 풀이. 이번 달 들어온 기운이 이 사람 일간·사주와 만나 어떤 흐름을 만드는지. 5~6문장.>",
  "weekly": [
    { "week": 1, "body": "<1주차 흐름. 3문장.>" },
    { "week": 2, "body": "<2주차 흐름. 3문장.>" },
    { "week": 3, "body": "<3주차 흐름. 3문장.>" },
    { "week": 4, "body": "<4주차 흐름. 3문장.>" }
  ],
  "sections": [
    { "key": "money",  "body": "<재물운. 5~6문장.>" },
    { "key": "work",   "body": "<직장·업무운. 5~6문장.>" },
    { "key": "love",   "body": "<애정·관계운. 5~6문장.>" },
    { "key": "health", "body": "<건강·멘탈운. 5~6문장.>" },
    { "key": "study",  "body": "<학업·문서·이동운. 5~6문장.>" }
  ],
  "timing": { "good": "<이번 달 흐름이 좋은 시기. 상순/중순/하순 정성적. 2문장.>", "caution": "<조심·점검할 시기. 2문장.>" },
  "balance": { "good": "<이번 달 하면 좋은 일 한 줄>", "warn": "<주의할 점 한 줄, 따뜻한 대비로>" },
  "note": "<별콩이의 한마디. 2~3문장.>"
}
```
규칙은 daily 와 동일(반말 친구 말투, 단정 금지·흐름/가능성, 좋기만 한 예언 금지, JSON 외 텍스트 금지, 문자열 내 `"` escape·줄바꿈 금지).

**분량:** `MAX_TOKENS_BY_FORTUNE.monthly` 2,560 → **5,000** (daily ×2.0). saju_full(8,192)보다 가벼움.

## 4. 생성 API — `app/api/fortune/create/route.ts`

- **월건 확보:** saju 기반이고 `cfg.type === "monthly"` 일 때 `saju.temporal = calcTemporalLuck(new Date(), sajuInput.year)` 호출(daily 와 동일, `temporal.month` 사용). daily 전용 분기에 monthly 추가.
- **monthly 파싱·병합·저장:** daily 분기와 평행하게, `cfg.type === "monthly"` 면 `parseMonthlyReportJson` → 실패 시 1회 재생성 → 그래도 실패면 `generation_failed`(502, 깨진 템플릿 저장 금지) → 성공 시 `serializeMonthlyReport(buildMonthlyReport(ai, saju.temporal))` 저장.
- **profile_id 저장:** 현재 `profile_id: null` 하드코딩 → 저장된 프로필 사용 시 **그 `profileId` 저장**(월간 per-profile 다시보기 판별용). 직접 입력(profileId 없음) 케이스는 null 유지.

## 5. 같은 달 다시보기 — 사주 선택 화면

- **`lib/fortune/monthly-lookup.ts` (신규):**
  - `kstMonthWindowUtc(now?)` — KST 기준 이번 달 1일 0시 ~ 다음 달 1일 0시를 UTC ISO 로.
  - `findThisMonthMonthlyByProfile(userId): Promise<Record<string, string>>` — 이번 달 monthly 리딩을 `profile_id` 별 최신 1건으로 그룹핑한 `{ profileId: readingId }` 맵. (profile_id 가 null 인 legacy 행은 제외.)
- **`GET /api/fortune/monthly-existing` (신규):** 로그인 필요. `{ existing: Record<profileId, readingId> }` 반환.
- **`FortuneSajuPicker` 신규 옵션 prop:**
  - `reviewableByProfile?: Record<string, string>` — profileId → 기존 리딩 id.
  - `onReview?: (readingId: string) => void`.
  - 선택된 프로필이 맵에 있으면 하단 CTA 라벨을 **"이번 달 운세 다시보기"** 로 바꾸고 클릭 시 `onReview(id)`. 없으면 기존 `onConfirm` 동작. (옵션 미전달 시 완전 무변경 — daily/saju_full 영향 없음.)
- **monthly `[type]` 페이지** (`type === "monthly"` 일 때만): 마운트 시 `/api/fortune/monthly-existing` fetch → `reviewableByProfile` + `onReview={(id) => router.push('/fortune/result?id=' + id + '&from=history')}` 전달. saju_full 등 다른 타입엔 미전달.

## 6. 결과 화면 — `app/fortune/result/page.tsx`

- `ft === "monthly"` 면 `tryParseStoredMonthlyReport(report)`:
  - 성공 → **`MonthlyReportView`** 렌더.
  - 실패(legacy 마크다운) → 기존 `parseSections` 폴백(현행 유지).
- 공유 텍스트: `monthlyReport` 있으면 `buildMonthlyShareText(report, label, url)` 분기(daily 의 `buildDailyShareText` 와 동형 — 별점·테마·월건·주차·도메인·시기·한마디·링크).
- 헤더 날짜: monthly 는 `createdAt` 에서 "2026년 6월"(연·월) 라벨.

**렌더 컴포넌트 구조** — 분량이 daily ×2 라 단일 컴포넌트 대신 시각적으로 구분되는 **카드 블록 5개**로 분할. `components/fortune/monthly/` 폴더로 묶어 daily 컴포넌트와 분리한다. 모두 DailyReportCard 톤 계승(라벤더/크림 팔레트, 오행 색 점).

- **`MonthlyReportView.tsx`** — props `{ report: MonthlyReport; monthLabel: string }`. 아래 5개 블록을 세로 간격(`space-y-4` 정도)으로 조합. 결과 페이지는 이 컴포넌트 하나만 렌더.
- **`MonthlyOverviewCard.tsx`** — 상단 히어로 요약. 월 라벨(2026년 6월) + 월건 두 글자 칩(한자 + 오행 색 점) + 테마 한 줄 + 종합운 별점(★×stars) + 총평(summary) + 행운 키워드/색 칩.
- **`MonthlyFlowCard.tsx`** — "이번 달 흐름" 묶음. 월건 풀이(intro) 문단 + **주차별 흐름** 1~4주 세로 타임라인(주차 마커 + body).
- **`MonthlyDomainsCard.tsx`** — 도메인 5개(아이콘 + 제목 + body). `DAILY_SECTIONS` 메타 재사용.
- **`MonthlyTimingCard.tsx`** — 짧은 콜아웃 묶음. **주목할 시기**(좋은 시기 / 조심 시기 2칸) + 이번 달 챙길 점(✅ good / ⚠️ warn).
- **`MonthlyNoteCard.tsx`** — 별콩이 한마디(note). 다크 테마 박스(DailyReportCard note 와 동일 톤).

## 에러 처리

- monthly JSON 파싱 1회 재시도 후 실패 → `generation_failed`(502), 저장 안 함(daily 와 동일 정책).
- `temporal.month` 없으면(이론상 saju 계산 실패) monthly 생성 실패 처리.
- 결과 화면에서 monthly 인데 JSON 파싱 실패(legacy/손상) → 마크다운 폴백으로 graceful.
- `monthly-existing` fetch 실패 → 맵 빈 객체로 처리(다시보기 미노출, 정상 결제 플로우). 비차단.

## 테스트 (순수 로직 우선)

- `parseMonthlyReportJson`: 정상 / 코드펜스 감쌈 / weekly 3개만(누락) / 도메인 4개만 / timing 누락 / stars 범위 밖(클램프) → 각 기대값.
- `tryParseStoredMonthlyReport`: v:1 JSON 왕복 / legacy 마크다운 → null / 손상 JSON → null.
- `buildMonthlyReport`: temporal.month → 월건 글자·오행 매핑 정확.
- `kstMonthWindowUtc`: 월말/연말 경계(12월 → 다음 해 1월).
- 수동 e2e: monthly 생성 → 카드 렌더 → 같은 달 재선택 시 "다시보기" CTA → 기존 리딩 이동 + 하단탭 내 고민톡 filled(from=history).
