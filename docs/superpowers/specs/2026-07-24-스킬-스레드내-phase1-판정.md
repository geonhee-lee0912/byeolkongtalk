# 스킬을 스레드 안으로 — Phase 1: 싸움 판정 (설계)

- 날짜: 2026-07-24
- 상태: 설계 합의 → 스펙 리뷰 대기
- 큰 그림: "우리 사이" 스킬을 **별도 페이지/별도 대화 없이 하나의 연애 상담 스레드 안에서** 제공하도록 재설계(대화 분산 제거). 스킬별 순차: **판정 → 궁합 → 카드뽑기**. 이 스펙은 **Phase 1 = 판정(verdict)** 만 다룬다.

## 배경 / 문제

현재 스킬은 별도 화면 + 별도 `readings`를 만든다. 특히 **판정**은 `/relationship/verdict/[id]`에서 **완전히 분리된 5턴 대화**로 진행 → 연애 상담의 대화가 여러 곳으로 **분산**된다. 유저가 스레드에서 "타로 뽑고싶어/판정받고싶어"라고 해도 별콩이는 "여긴 그거 없어"라고 하거나 딴 페이지로 튕겨 나간다. 목표는 **별콩이 = 하나의 연애 상담 에이전트**, 스킬은 그 대화 중에 쓰는 능력으로 만드는 것.

이동이 사라지면 이전 사이클에 만든 **복귀 인사 버블/결과 CTA/recap-seen** 은 존재 이유가 없어진다. 단 그 기능은 아직 **궁합·카드뽑기**(Phase 2·3 전까지 이동형 유지)가 쓰므로 **Phase 1에서는 제거하지 않는다** — Phase 3 완료 시 최종 정리.

## 목표 / 비목표

**목표 (Phase 1)**
- 판정을 스레드 안에서 진행: 확인 모달(30⭐) → 별콩이가 **같은 스레드에서** 양쪽 입장 듣고 비율 판정 + 화해 처방 → 일반 대화로 자연 복귀. 별도 페이지/별도 reading 없음.
- 판정 대화는 전부 스레드 메시지로 남아 분산 0.
- 별콩이가 스레드에서 타로/판정을 요청받으면 거부 대신 스킬로 연결(P1 페르소나 픽스, 판정 부분).

**비목표 (다음 Phase)**
- 궁합(Phase 2), 카드뽑기(Phase 3) 스레드화.
- 복귀 인사/CTA/recap-seen 계열 제거(Phase 3에서 일괄 — Phase 1엔 그대로 둠).

## 결정 로그 (브레인스토밍)

| 결정 | 선택 |
|---|---|
| 접근 | A — 판정을 `/api/relationship/chat`의 '모드'로(별도 라우트/reading 없음). 대안 B(verdict 라우트 살려 thread에 쓰기)는 라우트 이원화라 반려. |
| 과금 | 계속 30⭐ 유료(확인 모달) |
| 하루 캡 | 판정 세그먼트 턴은 일일 소프트캡(20턴)에서 **제외** (⭐가 게이트, 이중과금 방지) |
| 종료 신호 | `[SKILL_DONE]` 마커(관계 스레드는 `[END]` 금지라 별도) + 안전 턴캡 |
| 기존 판정 UI/라우트 | 제거(대체) |

## 설계

### 데이터

1. `relationships.memo`에 `active_skill` 추가 (JSONB, 마이그레이션 불필요):
   ```ts
   active_skill?: { key: string; started_at: string; assistant_turns: number } | null;
   ```
   진행 중 인-스레드 스킬 추적. `assistant_turns` = 판정 시작 후 별콩이 응답 턴 수(안전 턴캡용).

2. **`messages.skill_key`** 컬럼 추가 (마이그레이션, `VARCHAR(30) NULL`): 스킬 세그먼트 중 생성된 메시지 태깅. 하루 캡 계산이 `role='user' AND skill_key IS NULL`만 세도록 → 판정 턴 캡 제외. (Phase 2·3에서도 재사용하는 공통 토대.)
   - 마이그레이션: `supabase/migrations/<ts>_messages_skill_key.sql` — `ALTER TABLE messages ADD COLUMN skill_key VARCHAR(30);`

### 개시 (client → chat 라우트)

- `useSkillLaunch`: 판정(kind `dialogue`)은 지금처럼 확인 모달(30⭐)을 연다. 단 "확인하고 시작" 시 `launchDialogue`(POST verdict + 이동) 대신 **ThreadChat이 제공한 콜백 `onInThreadSkill("verdict")`** 호출.
- `ThreadChat.onInThreadSkill`: `/api/relationship/chat`에 **`skillStart: "verdict"`** 로 요청(유저 메시지 없음). 스트리밍으로 별콩이의 판정 도입을 받아 스레드에 붙인다.
- 차감은 **서버(chat 라우트 skillStart 처리)에서** `spendStars(userId, 30, source:"rel_skill_verdict")`. 잔액 부족이면 402 → 클라가 `/shop`. (확인 모달이 이미 클라 잔액 체크하지만 서버가 최종 권위.)

### 채팅 라우트 (`/api/relationship/chat`) 변경

- **`skillStart?: string`** 파라미터 수용:
  1. 활성 패스 확인. 이미 `active_skill` 있으면 400(중복 방지).
  2. `skillStart==="verdict"`면 `spendStars(30, "rel_skill_verdict")` (실패 시 402).
  3. `memo.active_skill = { key:"verdict", started_at:now, assistant_turns:0 }` 저장.
  4. 모델 입력 = DB 최근창 + **비영속 합성 트리거**(예: user "판정 시작") — 이 트리거는 저장하지 않음(스레드 오염 방지, 기존 verdict 페이지 KICKOFF 패턴 이식). 시스템 메시지에 판정 도입(§1단계) 가이드 주입 → 별콩이 도입 스트리밍.
  5. 별콩이 도입 응답만 assistant 메시지로 저장(`skill_key="verdict"`).
- **일반 턴 처리 시 `active_skill==="verdict"`면**:
  - 시스템 메시지에 **판정 가이드**(아래 페르소나) 주입(일반 관계 톤 대신 판정 진행).
  - user/assistant 메시지 저장 시 `skill_key="verdict"` 태깅 → **하루 캡 카운트 제외**.
  - **캡 게이트 우회**: `active_skill` 중이면 소프트캡 도달이어도 전송 허용(유료 세그먼트).
  - `assistant_turns`++; assistant 응답에 **`[SKILL_DONE]`** 있으면 → `active_skill=null`, **`skill_log`만 적립**(판정 요약, `cleanSummary`) — ⚠️ `pending_skill_recap`은 **세팅하지 않음**(인-스레드라 복귀 인사 불필요; `applySkillToMemo` 대신 skill_log에 직접 append). 일반 대화 복귀.
  - **안전 턴캡**: `assistant_turns >= 6`인데 마커 없으면, 이번 턴 시스템에 "이번 응답에서 반드시 판정 마무리 + `[SKILL_DONE]`" 강제 주입 후 종료.
- `getTodayThreadTurns`는 `role='user' AND skill_key IS NULL`만 카운트하도록 수정.

### 페르소나 (판정 인-스레드 가이드)

- 기존 `data/persona/byeolkong_verdict.md`는 **별도 시스템 프롬프트**(core+verdict, `[END]` 사용)라 그대로 못 씀. 인-스레드용으로 각색한 블록을 만든다:
  - `data/persona/byeolkong_verdict_inthread.md` (또는 `lib/claude.ts` 내 상수) — 3단계(입장 청취 → 비율 판정 → 화해 처방), **`[END]` 대신 `[SKILL_DONE]`**, 관계 스레드 톤 유지, "딴 판(사주/타로) 읽는 척 금지" 코어 규칙 준수.
- `buildRelationshipSystemMessage`가 `activeSkill` 인자를 받아 이 블록을 dynamicPart에 주입(없으면 일반).

### 페르소나 (P1 — 판정 요청 시 제안)

- `data/persona/byeolkong_relationship.md` §스킬 제안: 유저가 "누가 잘못한 건지 판정해줘" 류를 꺼내면 거부하지 말고 `[SKILL:verdict]` 제안(연속성 예고 규칙과 동일 톤). (타로 거부 일반 픽스는 Phase 3에서 카드뽑기와 함께 완성 — 판정 부분만 여기서.)

### 클라이언트 (ThreadChat)

- `useSkillLaunch`에 `onInThreadSkill?(key)` 추가; 판정은 이 콜백으로 분기(compat/tarot는 기존 그대로 — 아직 이동형).
- `ThreadChat`: `onInThreadSkill = (key) => sendSkillStart(key)` — `/api/relationship/chat`에 `{ relationshipId, skillStart:key }` POST, 스트리밍을 기존 send()와 동일하게 렌더.
- `active_skill` 중엔: (a) 다른 스킬 시작 막기(⚡ 시트/칩 비활성 또는 토스트), (b) 캡 도달이어도 입력 허용. `active_skill` 상태는 GET `/api/relationship` 응답에 포함해 클라가 알게 한다.

### 제거 (Phase 1 한정)

- `app/relationship/verdict/[id]/page.tsx`
- `app/api/relationship/verdict/route.ts`, `app/api/relationship/verdict/chat/route.ts`
- `lib/relationship/useSkillLaunch.ts`의 `launchDialogue`
- (복귀 인사/CTA/recap-seen·`pending_skill_recap`은 **유지** — 궁합·카드뽑기가 아직 씀. Phase 3에서 제거.)

## 흐름 (end-to-end)

```
연애 상담 스레드
  └─ (판정 칩/시트) → 확인 모달 30⭐
       └─ 확인 → chat 라우트 skillStart:"verdict" (30⭐ 차감, active_skill 세팅)
            └─ 별콩이(스레드): "무슨 일이 있었는지 얘기해줘 — 양쪽 다 듣고 판정해줄게" (skill_key=verdict, 캡 제외)
                 └─ 유저·별콩이 몇 턴(입장 청취) …
                      └─ 별콩이: 비율 판정 + 화해 처방 + [SKILL_DONE]
                           └─ active_skill 해제 · skill_log 적립 · 일반 대화 복귀 (전부 같은 스레드)
```

## 엣지 / 무결성

- **중복 시작**: `active_skill` 있으면 skillStart 400 + 클라가 스킬 칩 비활성.
- **미완료 이탈**: 유저가 판정 중 이탈 후 재방문 → `active_skill` 유지되어 이어서 진행(스레드라 자연). 안전 턴캡으로 무한 방지.
- **환불**: skillStart에서 차감 후 도입 스트리밍 실패 시 30⭐ 환불 + `active_skill` 롤백(기존 fortune 패턴 참고).
- **캡 상호작용**: 판정 턴은 `skill_key` 태깅으로 카운트 제외 → 판정 후에도 일반 대화 20턴 온전.

## 테스트

- 유닛: `getTodayThreadTurns`가 `skill_key IS NULL`만 세는지(쿼리 변경 — 통합/모킹). `active_skill` 세팅/해제 순수 로직(있으면).
- 라우트: skillStart가 차감+active_skill 세팅+도입 스트리밍; `[SKILL_DONE]`이 해제+skill_log; 중복 skillStart 400; 안전 턴캡.
- QA 하네스: 판정 세그먼트가 3단계로 진행되고 `[SKILL_DONE]`로 닫히는지(페르소나).
- 브라우저 E2E(dev): 칩→확인→스레드 내 판정→복귀, 캡 제외 확인.
