# 우리 사이 — 스킬 연속성 (설계)

- 날짜: 2026-07-23
- 상태: 설계 승인 → 구현 플랜 대기
- 범위: "우리 사이"(관계 스레드)에서 스킬 4종(관계 체크인 · 걔 속마음 · 우리 궁합 · 싸움 판정)을 소환했을 때, 스레드 대화와 **더 자연스럽게 이어지게** 만든다.

## 배경 / 문제

현재 스킬은 전부 **화면 전환** 방식이다. 소환하면 별도 화면으로 나가 별도 `readings` 레코드를 만든다:

- `checkin` / `deep_feelings` (`kind: tarot_draw`) → `/tarot/draw` → 타로 리딩
- `compat` (`kind: compat`) → `POST /api/fortune/create` → `/fortune/result`
- `verdict` (`kind: dialogue`) → `POST /api/relationship/verdict` → `/relationship/verdict/[id]`

결과가 스레드로 돌아오는 유일한 경로는 `logSkillToThread` → `relationships.memo.skill_log`(요약 120자) 적립이고, 이건 시스템 파일블록 `[지난 스킬: …]`으로 **수동적으로만** 주입된다. 그래서:

1. 소환 전: 유저가 스킬을 누르면 대화에서 튕겨 나가는데, "돌아와서 이어진다"는 신호가 없다.
2. 소환 후: 스레드로 돌아와도 별콩이가 결과를 **먼저 짚어주지 않는다**. 어드민에서 봐도 스킬은 스레드 대화의 턴이 아니라 독립 reading이라 흐름이 갈라져 보인다.

목표는 이 왕복을 **의도된 연속**으로 느끼게 하는 것. (인라인 렌더로 화면 전환 자체를 없애는 건 비목표 — 스킬별 리치 UI 재현 부담 + `verdict`는 자체 대화형이라 큰 공사.)

## 목표 / 비목표

**목표**
- (①) 스레드에서 별콩이가 스킬을 제안(`[SKILL:key]`)할 때, "결과 보고 오면 여기서 이어서 얘기하자"는 **연속성 예고**를 반드시 곁들인다.
- (②) 스킬 결과 화면에서 **한 번에 스레드로 돌아오는 동선**을 준다.
- (③) 돌아오면 별콩이가 **결과를 먼저 짚으며** 대화를 잇는다.
- 스킬 4종 전부에 적용.

**비목표 (이번에 안 함)**
- 스킬 결과를 스레드 안에 인라인 렌더 (화면 전환 유지).
- 소환 칩 하단 고정 서브라벨(모델 무관 보증 텍스트).
- 별콩이 응답 문장에 서버가 고정 문구 강제 삽입.
- 복귀 즉시 모델 호출(자동 발화) — 턴/패스 소모·402 위험 때문에 채택 안 함.

## 결정 로그 (브레인스토밍)

| 결정 | 선택 | 이유 |
|---|---|---|
| 연속성 수준 | 복귀 이어받기 + 복귀 동선 둘 다 | 인라인은 과함. 기존 복귀 장치 재사용이 저렴. |
| 적용 범위 | 스킬 4종 전부 | seed·예고는 공통 코드 한 곳이라 저렴. |
| ③ 이어받기 실현 | **복귀 즉시 템플릿 인사 버블** | 모델·턴·패스 소모 없음. skill_log가 이미 파일블록에 있어 이후 실제 턴은 자연 연결. |
| ① 예고 강제 | **페르소나 규칙만(MUST)** | 칩 서브라벨/서버강제 없이 자연스러운 말투 유지. |

## 설계

### ① 소환 예고 — 페르소나 규칙

- 파일: `data/persona/byeolkong_relationship.md` → `## 스킬 제안 ([SKILL:key])` 섹션.
- 규칙 추가(MUST): `[SKILL:key]` 마커를 낼 때, 마커 직전 문장에 **연속성 약속**을 한 문장 포함한다 — "갔다 오면 그 결과 갖고 여기서 바로 이어서 보자" 류.
- ⚠️ 같은 섹션 72행의 "탈출구·유도 문구 금지"와 충돌 주의: "안 봐도 돼 / 부담 없이" 같은 **여지를 주는 문구가 아니라**, 돌아와서 **이어진다는 약속**으로 써야 한다. (제안은 힘 있게, 다녀오면 이어짐을 확신시키는 톤.)

### ② 복귀 동선 — 결과 화면 → 스레드

- 스킬 reading(= `readings.relationship_id`가 있는 reading)의 결과 화면 3종에 CTA 추가: **"💬 우리 사이로 돌아가 이어 얘기하기"** → `/relationship`.
  - 궁합: `/fortune/result` (compat reading)
  - 카드뽑기(체크인·걔속마음): 타로 스킬 리딩/결과 화면
  - 판정: `/relationship/verdict/[id]`
- 노출 조건: `reading.relationship_id != null` (일반 리딩엔 안 뜸). 정확한 컴포넌트 위치는 구현 플랜에서 특정.

### ③ 복귀 즉시 템플릿 인사 버블 (무과금)

**데이터**
- `lib/relationship/types.ts` `RelationshipMemo`에 추가:
  ```ts
  pending_skill_recap?: { skill: string; summary: string; created_at: string } | null;
  ```

**seed (스킬 완료 시)**
- `lib/relationship/skill-log.ts` `logSkillToThread(relationshipId, skillKey, readingId, summary)`가 `skill_log` 적립과 **같은 업데이트**에서 `memo.pending_skill_recap = { skill: skillKey, summary, created_at }`도 세팅.
- `logSkillToThread`는 이미 3경로(fortune/create=compat, tarot/chat=카드뽑기, relationship/verdict/chat=판정)에서 호출되므로 **4종 공통 커버** (호출부 수정 불필요).

**surface (복귀 시)**
- `GET /api/relationship` 응답에 `recap`(= `memo.pending_skill_recap`) 포함.
- `app/relationship/page.tsx`가 `recap`을 `ThreadChat`에 프롭(`skillRecap`)으로 전달 — **활성 스레드(S3) 렌더에서만**.
- `components/relationship/ThreadChat.tsx`: 마운트 시 `skillRecap`이 있으면
  1. 템플릿 별콩이 버블을 화면 메시지에 1회 append (`useRef` 가드로 재렌더/재마운트 중복 방지). **비영속** — DB `messages`에 저장하지 않음.
  2. 소진 호출: `POST /api/relationship/recap-seen` (fire-and-forget) → 서버가 `memo.pending_skill_recap = null`.
- 버블 템플릿(클라 헬퍼): 스킬 label·emoji + summary 사용. 예: "💑 우리 궁합 결과 봤어! {summary} — 이거 갖고 더 얘기하고 싶은 거 있으면 편하게 꺼내봐." (문구는 구현 시 톤 다듬기.)
- 버블 이후: 유저의 다음 실제 턴부터는 파일블록 `[지난 스킬: …]` 주입(기존)으로 모델이 자연스럽게 이어감 — 추가 조치 불필요.

## 흐름 (end-to-end)

```
스레드 대화 중
  └─ 별콩이: "…그 얘기라면 우리 궁합으로 더 자세히 볼 수 있어. 갔다 오면 여기서 이어서 보자 [SKILL:compat]"   ← ①
       └─ (칩) 탭 → /fortune/result (compat reading 생성, logSkillToThread → skill_log + pending_skill_recap seed)   ← ③ seed
            └─ 결과 화면: "💬 우리 사이로 돌아가 이어 얘기하기"   ← ②
                 └─ /relationship 진입 → GET /api/relationship (recap 포함)
                      └─ ThreadChat: 템플릿 인사 버블 1회 + POST recap-seen(소진)   ← ③ surface
                           └─ 유저 다음 발화 → 모델이 [지난 스킬] 근거로 자연 연결
```

## 엣지 / 무결성

- **1회 소진**: `useRef` 렌더 가드 + 서버 소진(`recap-seen`). recap 내용은 `skill_log`에 영속되므로 드물게 이중 노출돼도 무해.
- **`pending_checkin`과 독립**: checkin은 다음 실제 턴에 별콩이가 먼저 안부(모델), recap은 복귀 즉시 UI 버블 — 서로 안 겹침. 둘 다 있어도 각자 동작.
- **무과금**: recap 버블은 모델 호출·턴·패스 소모 없음. 패스 만료(S2 읽기전용)에선 버블 렌더 안 함(S3에서만).
- **최신만**: 연속 스킬(A→B) 후 복귀 시 `pending_skill_recap`은 최신(B). 이전 것도 `skill_log`엔 남아 모델 컨텍스트로 유지.
