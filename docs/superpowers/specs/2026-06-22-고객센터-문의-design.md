# 고객센터 / 문의 기능 (사용자 ↔ 어드민 1:1 문의) — 설계

> 작성일 2026-06-22 · Phase 4(d) admin / 운영 도구 계열

## 1. 목적 / 배경

사용자가 서비스에 대한 불만·버그·환불·제안 등을 서비스 제공자(운영자)에게 직접 전달하고, 운영자가 어드민에서 그 답변을 작성·전송하는 1:1 문의 채널. 현재 `/mypage` 의 "고객센터 / 문의" 메뉴는 `alert("곧 열릴 예정")` stub 상태 — 이걸 실제 기능으로 교체한다.

## 2. 핵심 결정 (확정)

| 항목 | 결정 |
|---|---|
| 대화 구조 | **1회 Q → 1회 A** (단순). 문의 1건 = DB 1 row(질문 + 답변 컬럼). 무한 스레드 X |
| 카테고리 | **5개**: 버그 / 환불·결제 / 제안·건의 / 이용문의 / 기타 |
| 답변 알림 | **인앱 배지** — 안 읽은 답변이 있으면 BottomTab '내 정보' + mypage 메뉴에 🔴 |
| 첨부파일 | **텍스트만** (MVP). Storage 미사용 |
| 사용자 삭제 | **답변 전(`status='open'`)에만** 가능. 답변되면 기록 유지 |
| 어드민 배지 | **미답변(open) 건수** 를 사이드바 NAV에 표시 |
| 환불 연동 | **수동** — 상세 화면에 `/admin/payments` 링크만. 실제 환불은 기존 결제 어드민에서 |
| 인증 | 로그인 필수 — 세션 `user_id` 기준 (readings 와 동일). `/mypage` 가 이미 auth-gated |

## 3. 데이터 모델 — `inquiries` (단일 테이블)

`supabase/migrations/<timestamp>_inquiries.sql`:

```sql
CREATE TABLE IF NOT EXISTS inquiries (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  category    TEXT NOT NULL,        -- 'bug'|'refund'|'suggestion'|'usage'|'etc'
  title       TEXT NOT NULL,
  body        TEXT NOT NULL,
  status      TEXT NOT NULL DEFAULT 'open',   -- 'open' | 'answered'
  answer_body TEXT,                 -- 어드민 답변 (null = 미답변)
  answered_at TIMESTAMPTZ,
  answered_by UUID REFERENCES users(id) ON DELETE SET NULL,
  read_at     TIMESTAMPTZ,          -- 사용자가 답변을 확인한 시각 (배지 해제용)
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_inquiries_user   ON inquiries(user_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_inquiries_status ON inquiries(status, created_at DESC);
ALTER TABLE inquiries ENABLE ROW LEVEL SECURITY;  -- 기존 패턴: service_role 로만 접근, RLS 정책 미추가
```

- `category` 는 애플리케이션 레벨 enum 검증(아래 상수). DB CHECK 제약은 두지 않음 — 기존 테이블(`admin_actions` 등)이 주석 기반으로 관리하는 컨벤션 따름.
- `ON DELETE CASCADE` → 회원 탈퇴 시 문의 자동 정리 (`users → ...` 체인과 동일, withdraw 라우트 추가 코드 불필요).
- **안 읽은 답변 배지** = `status='answered' AND read_at IS NULL` row 개수.
- **미답변 어드민 큐** = `status='open'`, `created_at` 정렬.

### 카테고리 상수

`lib/inquiries.ts` (신규):

```ts
export const INQUIRY_CATEGORIES = {
  bug:        "버그 신고",
  refund:     "환불·결제",
  suggestion: "제안·건의",
  usage:      "이용 문의",
  etc:        "기타",
} as const;
export type InquiryCategory = keyof typeof INQUIRY_CATEGORIES;
```

## 4. 사용자 흐름 / UI (`/mypage/support`)

기존 `/mypage/payments` 서브페이지 패턴을 따름 (라우트 분리, client component, `/api/*` fetch).

- **`/mypage` 메뉴 교체** — 현 stub `alert` 제거 → `Link href="/mypage/support"`. 안 읽은 답변 있으면 메뉴 우측에 🔴 점.
- **`/mypage/support`** (목록) — 내 문의 카드 리스트: 카테고리 뱃지 · 제목 · 상태(`답변 대기`/`답변 완료`) · 작성일. 답변완료 & 미확인이면 카드에 🔴. 상단 "문의하기" 버튼. 빈 상태 안내.
- **`/mypage/support/new`** (작성) — 카테고리 셀렉트(5개) + 제목(input) + 내용(textarea). 길이 검증: 제목 1~80자, 내용 5~2000자 (클라이언트 + 서버 양쪽). 제출 성공 → 목록으로.
- **`/mypage/support/[id]`** (상세) — 상단: 내 질문(카테고리·제목·내용·작성일). 하단: **별콩이 톤 답변 카드**. 미답변이면 "별콩이가 확인하고 있어 ✨" 안내. `status='open'` 이면 "삭제" 버튼 노출. 진입 시 답변이 있고 `read_at IS NULL` 이면 `read_at` 마킹(배지 해제).

## 5. 어드민 흐름 / UI (`/admin/inquiries`)

기존 `/admin/sensitive` (검토 큐) 패턴과 동일.

- **사이드바 NAV 추가** — `app/admin/layout.tsx` `NAV` 배열에 `{ href: "/admin/inquiries", label: "문의/고객센터", emoji: "💬" }`. **미답변(open) 건수 배지** 표시 (NAV 항목 우측 숫자). 건수는 layout 또는 가벼운 fetch 로 조회.
- **`/admin/inquiries`** (목록) — 상태 필터(전체/미답변/답변완료) + 카테고리 필터 + 리스트(작성자 닉네임 · 카테고리 · 제목 · 상태 · 작성일). `status='open'` 우선, `created_at` 정렬.
- **`/admin/inquiries/[id]`** (상세 + 답변) —
  - 좌측: 문의 내용(카테고리·제목·내용·작성일) + **작성자 정보**(닉네임, user_id, 가입일, 별 잔액, 누적 결제 — `/admin/users/[id]` 의 조회 로직 재사용/참조).
  - 우측: 답변 textarea + "답변 전송" 버튼. 이미 답변된 건은 기존 답변 표시 + (선택) 재전송/수정.
  - **환불 카테고리(`refund`)** 면 `/admin/payments` 로 이동하는 안내 링크 노출 — 실제 환불은 거기서 수동 처리.

## 6. API 라우트

### 사용자 (로그인 필수, 세션 `user_id` 기준 소유권 검증)

| 메서드 · 경로 | 설명 |
|---|---|
| `GET /api/inquiries` | 내 문의 목록 |
| `POST /api/inquiries` | 작성 — category enum + title/body 길이 검증 |
| `GET /api/inquiries/[id]` | 내 문의 단건(소유권 검증). 답변 있고 미확인이면 `read_at` 마킹 |
| `DELETE /api/inquiries/[id]` | 내 문의 삭제 — 소유권 + `status='open'` 일 때만 허용 |
| `GET /api/inquiries/unread-count` | 배지용 — `status='answered' AND read_at IS NULL` 카운트 |

### 어드민 (`requireAdminWrite` + `logAdminAction` 기존 유틸 재사용)

| 메서드 · 경로 | 설명 |
|---|---|
| `GET /api/admin/inquiries` | 목록 + 작성자 조인 + 상태/카테고리 필터 |
| `GET /api/admin/inquiries/[id]` | 단건 + 작성자 정보 |
| `POST /api/admin/inquiries/[id]/reply` | 답변 저장(`answer_body`/`answered_at`/`answered_by`/`status='answered'`) + `logAdminAction` |

- `lib/admin-actions.ts` 의 `AdminActionName` union 에 `'inquiry_reply'` 추가. 감사 로그 `target_type: 'inquiry'`, `target_id: <inquiry id>`.
- 읽기(GET) 어드민 라우트는 `requireAdmin`, write(reply)는 `requireAdminWrite` (CSRF same-origin).

## 7. 배지 표시 구현

- **사용자**: `BottomTab` "내 정보" 탭 + `/mypage` 고객센터 메뉴 항목에 🔴. 클라이언트에서 `/api/inquiries/unread-count` 가벼운 fetch (마운트 시 1회 + 라우트 변경 시). count>0 이면 점 표시.
- **어드민**: NAV "문의/고객센터" 항목에 미답변 건수. `/admin/layout.tsx` 또는 작은 서버 조회로 count.
