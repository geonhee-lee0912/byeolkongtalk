# 궁합·관계 분석 (compat) 상품 설계

> 별콩 운세 신상품 — 두 사람 사주로 보는 **연애·결혼 궁합** 1종.
> 입력 → Claude 1회 → 구조화 JSON 리포트 1장. 대화형 아님.

**작성일:** 2026-06-07
**상태:** 설계 확정 (구현 대기)

---

## 1. 목표 한 줄

두 사람의 사주(일간·오행)로 **연애·결혼 궁합**을 정성 등급 + 5개 콘텐츠 섹션의 단일 스크롤 프리미엄 리포트로 제공한다.

## 2. 제품 결정 (확정)

| 항목 | 결정 |
|---|---|
| 관계 범위 | **연애·결혼 궁합 1종** (친구/가족/직장 분기 없음) |
| 입력 | 두 사람 모두 자유 — 저장 프로필 선택 **또는** 즉석 입력(저장됨) |
| 상단 후킹 | 점수(%) 아님 → **정성 등급 5단계** + 한 줄 테마 |
| 콘텐츠 | 오행 케미 · 끌림/성격 · 갈등 포인트 · 장기 전망 · 관계 조언 |
| 레이아웃 | **단일 스크롤** (서브탭 아님) |
| 데이터 배관 | **A안** — `saju_data`에 쌍 `{ a, b, names }` JSONB 저장 (스키마 변경 0) |
| 즉석 입력 | **저장함** — `POST /api/profiles`로 프로필 생성 후 그 id 사용 |
| 비용 | 20별 (`FORTUNE_CONFIG.compat.cost`, 이미 설정됨) |
| 토큰 | `MAX_TOKENS_BY_FORTUNE.compat` 2048 → **6000** 상향 |
| 시간 의존 | **없음** (saju_full처럼 타고난 사주 + 관계 자체. 일진/세운 미사용) |
| 중복 방지 | 없음 (daily/monthly 같은 dedup 불필요 — 매 호출 1회성 유료) |

### 정성 등급 5단계 (고정 enum)
AI가 아래 중 하나만 고름. 라벨별 색 매핑. 낮은 등급도 겁주지 않는 톤.
1. `천생연분`
2. `찰떡궁합`
3. `좋은 인연`
4. `서로 배우는 인연`
5. `노력이 필요한 인연`

## 3. 핵심 단순화: create API는 항상 두 profileId만 받는다

"즉석 입력도 저장"이므로, 입력 화면에서 즉석 입력 시 먼저 `POST /api/profiles`로 프로필을 만든 뒤 그 id를 쓴다.
→ create API의 compat 분기는 **profileA/profileB 두 개의 id**만 받으면 된다. 즉석입력용 `input` 분기 불필요. 이름은 프로필 `display_name`에서 가져온다.

## 4. 데이터 흐름

```
입력화면(/fortune/compat)
  ├─ 첫 번째 사람: 프로필 선택 OR 새로 입력→POST /api/profiles→id
  ├─ 두 번째 사람: 프로필 선택 OR 새로 입력→POST /api/profiles→id
  └─ [궁합 보기] → 별 차감 확인 모달
        ↓ POST /api/fortune/create { type:"compat", profileA, profileB }
서버
  ├─ 두 프로필 소유권 확인 + calcSaju ×2  (sajuA, sajuB)
  ├─ buildFortuneSystem("compat", { saju: sajuA, sajuB })
  ├─ generateOnce(...) → JSON 1개
  ├─ parseCompatReportJson → 실패 시 1회 재생성 → 그래도 실패 502
  ├─ serializeCompatReport(buildCompatReport(ai))  → messages.content
  └─ readings insert: saju_data = { a: sajuA, b: sajuB, names: {a,b} }, profile_id = profileA
결과화면(/fortune/result?id=...)
  ├─ GET /api/readings/[id] → sajuData = { a, b, names }
  ├─ tryParseStoredCompatReport(content) → CompatReport
  └─ <CompatReportView report saju={ {a,b,names} | null } />
        (공유 링크: public route가 sajuData=null → 사주판 숨김, 본문만)
```

## 5. 파일 구조

**신규**
- `lib/fortune/compat-report.ts` — 타입(`CompatReport`, `CompatGrade`) + `parseCompatReportJson` + `buildCompatReport` + `serializeCompatReport` + `tryParseStoredCompatReport`. saju_full 모듈 미러.
- `components/fortune/compat/CompatReportView.tsx` — 단일 스크롤 뷰.
- `components/fortune/DualSajuPicker.tsx` — 두 사람 슬롯 picker (프로필 선택 + 새로 입력 인라인 폼). 즉석 입력은 `ProfileForm` acquaintance 모드 + `POST /api/profiles` 재사용.
- `app/fortune/compat/page.tsx` — 입력 화면(`[type]` 동적 라우트는 단일 saju 가정이라 compat은 전용 페이지).

**수정**
- `lib/fortune/types.ts` — `FORTUNE_CONFIG.compat.active = true`; `MAX_TOKENS_BY_FORTUNE.compat = 6000`.
- `lib/fortune/prompt.ts` — `sajuBlock`에 두 번째 인물 블록 지원(또는 compat 전용 빌더 분기); `SECTION_GUIDE.compat` JSON 지시로 교체; `buildFortuneSystem` 시그니처에 `sajuB?` 추가.
- `app/api/fortune/create/route.ts` — compat 분기: profileA/profileB 로드·계산·`buildCompatReport` 저장·`saju_data` 쌍 저장.
- `app/fortune/result/page.tsx` — compat 파싱/렌더 분기 + 공유 텍스트 빌더 + 헤더 숨김 조건.
- `app/fortune/page.tsx` (랜딩) — compat active=true 라 '준비 중' 해제(자동).

## 6. 프롬프트 — compat JSON 출력 형식

`SECTION_GUIDE.compat` (saju_full 톤·규칙 동일, JSON 1개만 출력):

```json
{
  "grade": "<천생연분 | 찰떡궁합 | 좋은 인연 | 서로 배우는 인연 | 노력이 필요한 인연 중 정확히 하나>",
  "theme": "<두 사람 관계를 관통하는 한 줄 테마. 20자 내외.>",
  "summary": "<두 일간·오행이 만나 만드는 관계의 큰 그림. 따뜻한 핵심 요약. 3~4문장.>",
  "chemistry": "<오행 케미: A 일간과 B 일간(및 두 사주 오행)이 상생인지 상극인지, 그게 관계에 어떻게 작용하는지 쉽게 풀이. 5~6문장.>",
  "attraction": "<끌림·성격 케미: 서로 왜 끌리고 어떤 점이 잘 맞는지. 4~5문장.>",
  "conflict": "<갈등 포인트: 부딪히기 쉬운 지점·온도차. 겁주지 말고 '이럴 땐 이렇게 이해하면 좋아' 톤. 4~5문장.>",
  "longterm": "<장기 전망: 시간이 지나며 관계가 어떻게 흐를지, 결혼·장기 관점. 4~5문장.>",
  "advice": ["<관계를 위한 실천 1>", "<2>", "<3>"],
  "note": "<별콩이의 한마디. 두 사람에게 건네는 따뜻한 응원 2~3문장.>"
}
```

**규칙**: 반말 친구 말투, 단정 금지(흐름·가능성), 좋기만 한 예언 금지, `advice` 정확히 3개, `grade`는 enum 5개 중 하나, JSON 문자열 내 큰따옴표 escape·줄바꿈 금지.

**프롬프트 사주 블록**: 기존 `sajuBlock(saju)`는 1인용. compat은 두 인물을 `[첫 번째 사람 사주판]` / `[두 번째 사람 사주판]`으로 각각 제시(이름 라벨 포함)하고, "두 일간이 만났을 때의 관계가 이 리포트의 핵심"이라는 안내를 덧붙인다.

## 7. 뷰 — CompatReportView (단일 스크롤)

saju_full 뷰의 토큰·다크 카드 패턴 재사용. 위→아래:

1. **다크 히어로** (`DARK_GRADIENT`) — 등급 배지(라벨별 색) + 테마 + summary.
2. **두 사주판 나란히** — `SajuBoard showDetail={false}` ×2, 각 상단에 이름. 두 일간 비교 한 줄(예: `갑목 ↔ 병화`). saju 없으면(공유) "공유 링크에서는 사주판이 표시되지 않아".
3. **오행 케미 카드** — `chemistry` (강조 카드).
4. **섹션 카드 3개** — 끌림/성격(`attraction`) · 갈등 포인트(`conflict`) · 장기 전망(`longterm`).
5. **관계 조언** — `advice` 3개 번호 리스트.
6. **다크 별콩이의 한마디** — `note`.

색: `cream-warm` / `lilac-deep` / `lilac-mid` / `gold` / 본문 `text-[#322E3D]` (saju_full과 동일, `text-text` 미사용).

## 8. API 변경 상세

### `POST /api/fortune/create` (compat 분기)
- body: `{ type:"compat", profileA: string, profileB: string }`.
- 두 id 모두 `user_profiles`에서 `user_id` 소유권 확인. 없으면 404 `profile_not_found`.
- `profileA === profileB` 면 400 `same_profile` (같은 사람끼리 궁합 방지).
- `calcSaju` ×2. `saju.temporal` 미설정(시간 의존 없음).
- 비용 20별: 기존 잔액 사전 확인 + `spendStars` 로직 그대로(`source: "fortune_compat"`).
- 저장: `saju_data = { a: sajuA, b: sajuB, names: { a: nameA, b: nameB } }`, `profile_id = profileA`, `emotion_tag = "fortune:compat"`, `stars_spent = 20`.
- 파싱 실패 1회 재생성 → 실패 502 `generation_failed` (saju_full과 동일 가드).

### 결과 화면
- `ft === "compat"` → `tryParseStoredCompatReport(content)`; `sajuData`는 `{a,b,names}` 또는 null.
- 헤더 숨김·뒤로가기 버튼은 saju_full과 동일 패턴.
- 공유 텍스트: `buildCompatShareText(report, label, url)` — 등급 + 테마 + summary 발췌.

## 9. 범위 밖 (YAGNI)
- 친구/가족/직장 등 관계 타입 분기 — 연애·결혼 1종만.
- 궁합 점수(%)·행운 블록(color/direction 등).
- 타로 기반 궁합 — 사주 기반만.
- 같은 쌍 중복 방지(dedup)·다시보기 — 매번 새 리포트.
- 3명 이상 궁합.
