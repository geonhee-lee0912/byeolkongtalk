# 운세 가격 재책정 + 오늘의 타로 무료 횟수 표시 — 설계

**작성일:** 2026-06-07
**대상 프로젝트:** byeolkong_talk

## 목표

1. **Part 1** — 오늘의 타로(`tarot_daily`)에도 무료 잔여/전체 횟수(`N/5회`)를 랜딩에 표시한다. 오늘의 운세(`daily`, 사주)와 동일한 방식.
2. **Part 2** — 분량·상품 구성 변경을 반영해 운세 상품 가격을 전반 인상한다. 기준: **체감 가치·티어**, 별 충전 패키지는 **고정**(상품 가격만 조정).

## 배경

- 마지막 가격 논의 이후 상품 구성이 바뀜: 타로 5종(데일리+도메인 4종) 확장, 사주분석(50별) 추가, 궁합 2종(연애/인간관계). 또한 직전 작업으로 **3장 타로 리포트 분량이 늘어남**(카드별 5~6줄 + 종합 5~6문장×2문단).
- 가격은 `lib/fortune/types.ts`의 `FORTUNE_CONFIG`가 단일 출처. create 라우트가 `effectiveCost = cfg.cost`(무료 한도 시 `cfg.paidCost`)로 읽고, 랜딩·입력 화면·shop·OG 카드 모두 `FORTUNE_CONFIG`를 참조. 하드코딩된 가격 상한 없음.
- 별 충전 단가(`lib/constants.ts`): 10별=1,000원 / 30별=2,800원 / 70별=5,900원 / 150별=11,000원 / 300별=19,900원 → 별당 약 66~100원(중간값 80원). **이번 작업에서 변경하지 않음.**

## Part 1 — 오늘의 타로 무료 횟수 뱃지

### 현재 동작

`app/fortune/page.tsx`는 `/api/fortune/daily-status`만 fetch한다. 무료 뱃지는 `f.cost === 0`일 때 노출되며, 잔여 횟수(`무료 N/limit회`) / 소진 표기(`무료 소진 · ⭐ nextCost`)는 **`f.type === "daily"`인 경우에만** 채워진다. 따라서 `tarot_daily`는 `f.cost === 0`이라 "무료"라는 글자만 보이고 횟수가 없다.

### 변경

- **파일:** `app/fortune/page.tsx` (단일 파일). API·DB 변경 없음.
- `/api/fortune/tarot-daily-status`(이미 존재, `{ used, limit, remaining, nextCost }` 반환)를 `daily-status`와 **병렬로 fetch**한다.
- 두 무료 상품의 상태를 타입별로 조회할 수 있게 보관한다(예: `daily` 상태와 `tarot_daily` 상태를 각각 state로 두고, `f.type`으로 매칭).
- 무료 뱃지 렌더 로직을 일반화: `f.type`이 `daily` **또는** `tarot_daily`이고 해당 상태가 로드됐을 때
  - 잔여 > 0: `무료 {remaining}/{limit}회` (기존 골드 톤 뱃지)
  - 잔여 ≤ 0: `무료 소진 · ⭐ {nextCost}` (기존 회색 톤 뱃지)
- 상태가 아직 로드 전이면 기존처럼 `무료`만 표기(깜빡임 방지).
- 비-무료 상품(타로 단품·궁합·이번달·사주분석)은 변경 없음 — `⭐ {cost}`.

### 데이터 흐름

```
FortunePage mount
  → fetch /api/fortune/daily-status        → dailyStatus  state
  → fetch /api/fortune/tarot-daily-status  → tarotDailyStatus state
랜더 시 f.type === "daily" → dailyStatus, f.type === "tarot_daily" → tarotDailyStatus 로 뱃지 표기
```

### 엣지/에러

- 비로그인: 두 status API 모두 `remaining = limit`(=5)을 반환 → `무료 5/5회` 표기. 기존 daily 동작과 동일.
- fetch 실패: 해당 status는 `null` 유지 → `무료`만 표기(현행 폴백과 동일).

## Part 2 — 가격 재책정 (A안: 5 / 20 / 30 / 60)

### 티어 및 신규 가격

| 티어 | 상품(type) | 필드 | 현행 | 신규 |
|---|---|---|---|---|
| 데일리(무료 funnel) | `daily`, `tarot_daily` | `paidCost` | 5 | **5 (유지)** |
| 심층 단품 | `monthly` | `cost` | 15 | **20** |
| 심층 단품 | `tarot_love`, `tarot_money`, `tarot_career`, `tarot_relation` | `cost` | 15 | **20** |
| 궁합 | `compat`, `compat_social` | `cost` | 20 | **30** |
| 프리미엄 | `saju_full` | `cost` | 50 | **60** |

- 인상폭: 심층 단품 +33%, 궁합 +50%, 프리미엄 +20%. 데일리 무료 모델(무료 5회 → 소진 후 5별)은 유지.
- KRW 환산(중간값 80원/별): 단품 ~1,600원 · 궁합 ~2,400원 · 사주분석 ~4,800원.

### 변경

- **파일:** `lib/fortune/types.ts`의 `FORTUNE_CONFIG` (단일 파일). 마이그레이션·신규 컴포넌트 없음.
- 위 표대로 6개 항목의 `cost` 수정(`daily`/`tarot_daily`는 변경 없음).
- 단일 출처이므로 다음이 자동 반영됨:
  - 랜딩 `app/fortune/page.tsx` 가격 뱃지(`f.cost`)
  - 각 입력 화면(예: `TarotInput`의 `cfg.cost`/`cfg.paidCost`)
  - create 라우트 과금/환불(`effectiveCost = cfg.cost`)
  - shop/결과 OG 카드 등 `FORTUNE_CONFIG` 참조부

### 검증

- `npx tsc --noEmit` 통과.
- `npm run build` 통과.
- 스폿 체크: 타로 단품 입력 화면(`무료`가 아닌 `⭐20별`), 궁합 입력(`⭐30별`), 사주분석(`⭐60별`)이 새 값으로 표기되는지. create 라우트가 새 `cost`로 차감하는지(코드 경로상 `cfg.cost` 직접 사용이므로 자동).
