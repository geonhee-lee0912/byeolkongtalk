# 고민 상담 대화 — 끊어 보내기 대응 + 부재 감지 멘트

작성일: 2026-05-31
대상 파일: `app/tarot/reading/page.tsx` (거의 클라이언트 한정), 서버 무수정 예정

## 1. 문제

유저가 채팅을 한 번에 안 보내고 여러 말풍선으로 끊어 입력하는데, 현재는 대응이 안 됨.

### 현재 동작 (root cause)
`handleSubmit` → 즉시 `sendMessage`로 Claude 호출. 스트리밍이 시작되면 textarea·전송 버튼이 `disabled`.

그래서 끊어 입력하는 유저에게는:
1. 첫 조각 전송 → 별콩이가 **즉시** 미완성 맥락에 반응 시작
2. 그 순간 입력창 잠김 → 다음 조각 못 보냄
3. 별콩이 답 끝날 때까지 기다렸다 이어 입력 → 턴 추가 소모

결과: ① 반쪽 맥락에 응답 ② 턴/비용 낭비 + [END] 수렴 카운트 조기 소진 ③ 대화 리듬 깨짐.

## 2. 목표

- 유저가 말풍선을 연속으로 보낼 수 있고, 잠깐 멈추면 그동안 쌓인 조각을 **합쳐서 별콩이가 한 번만** 응답.
- 유저가 다음 줄을 **타이핑하는 중**이면 절대 먼저 응답하지 않음 (가장 중요한 안전장치).
- 유저가 자리를 비우면(10초+ 무활동) 별콩이가 먼저 안부 멘트로 말 걸어서 "사람 같은" 느낌.
- 비용·턴 카운트 오염 없음.

## 3. 핵심 설계

### 3.1 활동 기반 debounce (끊어 보내기 묶음)

타이머를 "전송 시점"이 아니라 **"마지막 활동 시점"** 기준으로 잰다. 활동 = 키 입력(`onChange`) 또는 전송(`handleSubmit`).

**플러시(별콩이 응답) 발동 조건 — 셋 다 만족:**
1. 대기 조각(아직 별콩이에게 안 넘긴 user 조각)이 1개 이상
2. 입력창이 비어있음 (반쯤 쓰다 만 게 없음)
3. 마지막 활동 후 `DEBOUNCE_FLUSH_MS`(= 2000ms) 경과

흐름 예시:
- "나 요즘 힘든데" 전송 → 대기조각=[1], 플러시 타이머 2s 시작, 화면에 user 버블 즉시 표시
- "남자친구랑…" 타이핑 시작 → 키 입력마다 타이머 리셋 → 타이핑 동안 안 쏨
- 중간에 5초 멈춰도 입력창에 글자 있음(조건 2 위반) → 안 쏨
- 마저 써서 전송 → 대기조각=[1,2], 입력 비고 타이머 재시작
- 2s 무활동 + 입력 빔 → 조각 [1]+[2]를 **줄바꿈으로 합친 하나의 user 메시지**로 별콩이 1회 응답

### 3.2 표현 방식 (화면 vs 모델 vs DB)

- **화면:** 보낸 대로 user 말풍선 여러 개 그대로 유지 (끊어 보낸 느낌 보존).
- **모델/API:** 조각들을 `\n`으로 이어 **하나의 user 메시지**로 전송 (Claude는 user/assistant 교대 요구 → 합치는 게 맞음).
- **DB:** 서버는 지금처럼 마지막 user 메시지 1행 + assistant 1행만 INSERT → 합쳐진 한 행으로 저장.
- **턴 카운트/[END] 수렴:** 서버가 DB 행 기준으로 세므로, 조각 N개가 합쳐 1턴 → 지금 로직 그대로 정상 동작.
- **이어하기(resume) 시 caveat:** DB엔 합쳐진 한 행이라, 라이브에선 N개 버블로 보였던 게 resume 땐 줄바꿈 포함 1개 버블로 보임. 드물고 영향 적어 허용.

구현 메모: 대기 묶음 시작(대기조각 0→1) 시점에 `baseHistoryRef = 현재 messages 스냅샷`을 잡아두고, 플러시 때 `apiHistory = [...baseHistoryRef(ephemeral 제외), { role:"user", content: 조각.join("\n") }]` 로 구성. 화면 `messages`는 N개 버블 유지, API body만 합쳐 보냄.

### 3.3 부재 감지 멘트 (presence nudge)

별콩이 응답이 끝나 공이 유저에게 넘어왔는데 유저가 가만히 있을 때, 별콩이가 먼저 말 건다.

- **방식:** 준비된 멘트 3~4개 중 랜덤. **Claude 호출 안 함 / DB 저장 안 함 / 턴 카운트 제외** (ephemeral, 클라이언트 표시 전용).
- **타이밍:** 2단계 후 종료.
  - 1단계: 응답 종료 후 `IDLE_NUDGE_1_MS`(= 10000ms) 무활동 → 멘트 1회
    - 예: "어디 갔어~? 천천히 생각해도 괜찮아 :)" / "음, 아직 거기 있어? 별콩이 여기서 기다릴게" / "다른 거 하는 중이야? 돌아오면 마저 봐줄게"
  - 2단계: 그래도 계속 무활동이면 1단계로부터 추가 `IDLE_NUDGE_2_MS`(= 40000ms) 후 더 부드러운 멘트 1회
    - 예: "별콩이 여기 있을게, 천천히 와" / "급할 거 없어. 마음 정리되면 다시 얘기하자"
  - 2단계 후 멈춤 (더는 안 건다).
- **발동 게이트:** 대기조각 0 AND 입력창 빔 AND not streaming AND not ended AND 별콩이가 최소 1회 응답함.
- **표현:** ephemeral 플래그 단 assistant 메시지로 `messages`에 끼워 넣어 정상 순서로 렌더. 모든 API body 구성 시 ephemeral 제외. DB INSERT 대상 아님.
- **유저 복귀 시:** 멘트는 화면에 그대로 남김(실제 일어난 대화처럼). idle 단계 카운터만 0으로 리셋.

## 4. 타이머 상태 머신

상수:
```
DEBOUNCE_FLUSH_MS = 2000
IDLE_NUDGE_1_MS   = 10000
IDLE_NUDGE_2_MS   = 40000   // 1단계 이후 추가 대기
```

리셋/전이 규칙:
- **활동(키 입력 or 전송):** idle 타이머 해제 + idle 단계 0으로 리셋. 대기조각 > 0 이면 플러시 타이머 (재)무장.
- **별콩이 응답 종료(`finishMessage`) & not ended:** (이 시점 대기조각 0) idle 1단계 타이머(10s) 무장.
- **스트리밍 시작(`sendMessage` 진입):** 모든 idle/플러시 타이머 해제.
- **isEnded:** 모든 타이머 해제, nudge 없음.

플러시 타이머와 idle 타이머는 "대기조각 유무"로 상호 배타 (대기조각 있으면 플러시, 없으면 idle).

## 5. 영향받는 함수 (클라이언트)

- `Message` 타입: `ephemeral?: boolean` 추가.
- 신규 ref: `pendingFragmentsRef`(string[]), `baseHistoryRef`(Message[]), `messagesRef`(messages 미러), `flushTimerRef`, `idleTimerRef`, `idleStageRef`.
- `handleSubmit`: 즉시 호출 대신 → 조각을 `pendingFragmentsRef`에 push + user 버블 즉시 `messages`에 append + (대기 0→1이면 `baseHistoryRef` 스냅샷) + 입력 비우기 + 플러시 타이머 재무장. (직접 fetch 안 함)
- 신규 `flushPending()`: 게이트 통과 시 조각 합쳐 `apiHistory` 구성 → `sendMessage(apiHistory, rid, { skipSetMessages:true })` → 대기 비우기.
- `sendMessage`: API body 구성 시 `ephemeral` 제외 필터 추가. `skipSetMessages` 옵션 추가(플러시 경로는 화면 이미 갱신됨이라 `setMessages` 생략). 그 외 동작 동일.
- `finishMessage`: 종료 후 not ended면 idle 1단계 타이머 무장.
- `handleFinish`(대화 마무리): 대기조각 + 현재 입력창 내용까지 합쳐 flush한 뒤 `forceEnd`로 전송. 타이머 해제.
- `onChange`: 기존 동작 + 활동 리셋(타이머 규칙대로).
- 신규 `pushNudge(stage)`: ephemeral assistant 멘트를 `messages`에 append.
- 언마운트 cleanup에 신규 타이머 해제 추가.

## 6. 범위 밖 (YAGNI)

- 별콩이 응답 **중** 끼어들기/큐잉 (스트리밍 중엔 지금처럼 입력 잠금 유지).
- 부재 멘트를 Claude가 매번 생성 (비용·턴 제외 처리 복잡 → canned 랜덤으로 충분).
- "2초 안 기다리고 즉시 보내기" 수동 버튼 (단순화 위해 제외, 필요 시 추후).
- 서버 라우트 변경 (합쳐진 메시지를 그대로 받으면 됨).

## 7. 트레이드오프 (명시)

- 모든 follow-up에 최대 2초 지연이 생김 (한 번만 보내는 유저도 2s 후 응답 시작). debounce 방식의 의도된 비용이며 2000ms는 균형값. 상수로 빼서 튜닝 가능.
- resume 시 끊어 보낸 N버블이 1버블로 합쳐 보이는 표시 불일치 (3.2 caveat). 드물고 영향 적음.

## 8. 검증 기준

- 끊어 3번 전송 후 멈춤 → 별콩이가 **1회만** 응답 (3개 합친 맥락).
- 전송 후 다음 줄을 천천히(>2s) 타이핑 → 타이핑 동안 응답 **안 나감**.
- 입력창에 글자 남긴 채 5s 정지 → 응답 **안 나감**.
- 응답 종료 후 10s 무활동 → 부재 멘트 1회. 추가 40s 무활동 → 멘트 1회 더. 그 후 멈춤.
- 부재 멘트 등장 후에도 [END] 턴/글자 카운트 영향 없음 (DB 미저장 확인).
- "대화 마무리" 시 대기 조각·입력창 내용 누락 없이 합쳐 전송 + [END].
