# 어드민 콘솔 (v2) — 설계

> Phase 4 (d). v1 `tarot-friend` 어드민 5개 섹션을 v2 스키마로 이식하고,
> v2 신규 기능(사주·타로·운세·프로필·별거래)을 전부 조회·관리한다.
> dev/prod는 환경별로 분리 운영한다.

작성일: 2026-06-21

---

## 1. 목표 / 비목표

**목표**
- v1 어드민(대시보드·사용자·민감·에러·결제)을 v2 스키마로 이식.
- v2 신규 기능 전부를 어드민에서 조회 + (선별적) 관리.
- dev/prod 환경별 분리 어드민.

**비목표**
- 통합 대시보드(한 화면에서 dev/prod 토글) — 채택 안 함.
- v1 `free_today_card_grant`(오늘카드 무료) — v2엔 해당 기능 없음.
- 차트/그래프 시각화 — 숫자 KPI까지만, 차트 라이브러리 도입 X.
- 고객센터/문의 섹션 — 마이페이지 문의 기능 미구현. 본 빌드 제외(§8 확장 포인트만).
- 운세 재량 환불 — 운세 환불은 시스템 자동. 어드민 수동 환불 버튼 없음(CS 보상은 별 잔액 조정으로 처리).

## 2. 아키텍처 — 환경별 분리 + 인증

- `/admin/*`를 **양쪽 배포에 동일 코드로** 배치. 각 배포는 자기 환경 Supabase만 가리킨다(상대 환경 키를 주입하지 않음 → 보안 최상).
  - prod 배포(byeolkongtalk.com/admin) → prod DB
  - dev 배포(dev.byeolkongtalk.com/admin) → dev DB
  - "양쪽 관리" = 각 URL로 각각 로그인.
- **인증**: 기존 `lib/admin.ts`의 `assertAdmin`(화이트리스트 `ADMIN_USER_IDS` + HMAC 토큰 `byeolkong_admin_token`) **그대로 재사용**. 이미 카카오 로그인 시 admin 토큰 발급됨.
  - `app/admin/layout.tsx`에서 서버 가드: 미인증 시 `/`로 redirect.
  - 모든 `/api/admin/*` 라우트도 `assertAdmin` 가드 + write는 Origin 검증(CSRF).
- **DEV/PROD 환경 배너**: layout 상단에 환경 배지. `process.env.VERCEL_ENV`(production/preview) 또는 base URL 기준. **prod=빨강 "PROD", dev=노랑 "DEV".** 화면이 동일하므로 오조작 방지 필수.
- **레이아웃**: v1의 다크 사이드바 패턴 재사용, 컬러만 별콩 톤으로. `robots: noindex`.

## 3. 섹션 구성 (사이드바 네비)

| 섹션 | 소스 | 조회 | 액션(write) |
|---|---|---|---|
| 📊 대시보드 | 집계 | 오늘/주간 KPI(가입·리딩·매출·미검토 민감·미해결 에러) | — |
| 👥 사용자 | `users`, `star_balances`, `user_profiles` | 목록/검색, 잔액, 생일 프로필, 리딩수 | **별 잔액 조정**, **무료 운세 횟수 부여** |
| 🔮 리딩/상담 | `readings`, `messages` | 사주·타로·운세 통합 목록(타입/유무료/날짜 필터) + 대화 상세 | **리딩·메시지 삭제** |
| 💳 결제/정산 | `payments` | 토스 결제 내역, 상태별 | **결제 환불/취소** |
| 🎁 운세 환불 | `fortune_refund_notices` | 운세 생성 실패 자동환불 모니터링(발생·환불 별·확인 여부) | — (조회 전용) |
| 🚑 민감 알림 | `sensitive_alerts` | 카테고리·심각도별, 미검토 우선 | **검토 마킹**(reviewed_at + action_taken) |
| 🚨 에러 로그 | `error_logs` | 미해결 우선, 심각도 | **resolve 마킹** |

리딩/상담은 `readings` 한 테이블을 `consultation_type`(saju/tarot) + `emotion_tag`(운세 종류) + `stars_spent`(0=무료) 필터로 본다. 사주/타로/운세를 별 섹션으로 쪼개지 않는다.

## 4. 스키마 변경 (신규 마이그레이션 2개)

### 4.1 `admin_actions` — 감사 로그
모든 write 액션을 기록(잔액조정·환불·삭제·검토마킹·무료부여 추적).
```
admin_actions (
  id UUID PK,
  admin_id UUID REFERENCES users(id),
  action TEXT,            -- 'star_adjust' | 'payment_refund' | 'reading_delete' | 'sensitive_review' | 'error_resolve' | 'fortune_grant'
  target_type TEXT,       -- 'user' | 'payment' | 'reading' | 'sensitive_alert' | 'error_log'
  target_id TEXT,
  payload JSONB,          -- 액션별 상세(조정량/사유 등)
  created_at TIMESTAMPTZ DEFAULT now()
)
```
service_role 전용 RLS.

### 4.2 `fortune_free_grants` — 무료 운세 횟수 부여
무료 잔여는 저장 카운터가 아니라 `limit - used`(used = 무료 리딩 count)로 파생된다.
횟수를 늘리려면 per-user 보너스가 필요하다.
```
fortune_free_grants (
  id UUID PK,
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  fortune_kind VARCHAR(40),   -- emotion_tag 센티넬 (daily / tarot_daily / ...)
  bonus_count INTEGER NOT NULL,
  granted_by UUID REFERENCES users(id),
  reason TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
)
```
**기존 코드 수정**: `/api/fortune/daily-status`, `/api/fortune/tarot-daily-status`(및 freeLimit 있는 운세)의 잔여 계산을
`remaining = (limit + Σbonus) - used`로 변경.

## 5. 공통 안전장치

- 모든 write 액션 = **확인 모달**(기존 `StarConfirmModal` 패턴 재사용) + `admin_actions` 기록.
- 결제 환불은 `lib/toss` cancel **멱등 처리**(중복 클릭/재시도 방어).
- 별 잔액 조정은 기존 `spend_stars`/`charge_stars` RPC 경유(트랜잭션 + audit 유지). 직접 UPDATE 금지.
- 리딩 삭제는 `messages` CASCADE 영향 고지 + 확인.

## 6. API 라우트 (신규)

```
app/api/admin/
  stats/route.ts                         GET   대시보드 KPI
  users/route.ts                         GET   목록/검색
  users/[id]/route.ts                    GET   상세(잔액·프로필·리딩수)
  users/[id]/stars/adjust/route.ts       POST  별 잔액 조정
  users/[id]/fortune-grant/route.ts      POST  무료 운세 횟수 부여
  readings/route.ts                      GET   리딩 목록(필터)
  readings/[id]/route.ts                 GET/DELETE  상세 / 삭제
  payments/route.ts                      GET   결제 내역
  payments/[id]/refund/route.ts          POST  결제 환불/취소
  fortune-refunds/route.ts               GET   자동환불 모니터링(조회 전용)
  sensitive/route.ts                     GET   민감 알림
  sensitive/[id]/review/route.ts         POST  검토 마킹
  errors/route.ts                        GET   에러 로그
  errors/[id]/resolve/route.ts           POST  resolve 마킹
```

## 7. 구현 순서 (한 spec, 단계 구현)

1. 인증·레이아웃·DEV/PROD 배너 + 대시보드(stats)
2. 사용자(목록·상세·별 잔액 조정·무료 횟수 부여) + 마이그레이션 2개 + status 라우트 수정
3. 리딩/상담(통합 목록·필터·대화 상세·삭제)
4. 결제/정산(환불) + 운세 환불 모니터링(조회)
5. 민감 알림·에러 로그(검토/resolve 마킹)

## 8. 향후 확장 포인트 — 고객센터/문의 (이번 빌드 제외)

마이페이지 문의 기능이 생기면 어드민에 문의 섹션을 끼울 수 있도록 설계를 막지 않는다.
- 예상 테이블: `inquiries(id, user_id, subject, body, status, created_at)` + `inquiry_replies` 또는 `reply_body`.
- 어드민: 사이드바 nav에 `💬 문의` 슬롯 추가, 목록·상세·**답변 작성**(write).
- 답변 기록은 `admin_actions`(action='inquiry_reply') 재사용 가능.
- 현재 구조에서 추가 비용: nav 한 줄 + 라우트 한 묶음. 본 설계가 이를 막는 결정은 없음.
